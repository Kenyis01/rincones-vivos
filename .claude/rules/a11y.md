# Accesibilidad (a11y)

Leer al construir componentes interactivos, formularios o navegación. Reglas **preventivas**. Para auditorías, corré una herramienta de accesibilidad (axe-core u otra) sobre el sitio servido. Objetivo: WCAG 2.1 AA.

## La base es HTML semántico
La mayor parte de la accesibilidad se resuelve con HTML correcto (ver `html-semantics.md`). Un `<button>` real ya es accesible por teclado; un `<div onclick>` no. Empezá siempre por el elemento correcto.

## Teclado
- Todo lo interactivo debe ser usable con teclado (Tab, Enter, Espacio, Escape).
- Usar elementos nativos (`<button>`, `<a>`, `<input>`) que ya traen el comportamiento.
- **Foco visible siempre.** Nunca eliminar el outline sin reemplazarlo por un indicador claro (`focus-visible:`).
- Orden de tabulación lógico (sigue el orden del DOM; no reordenar con CSS de forma confusa).

## Imágenes y medios
- `alt` significativo en imágenes informativas; `alt=""` en decorativas.
- No transmitir información solo con color (agregar texto/ícono).

## Formularios
- Cada campo con `<label>` asociado (`for`/`id`). El placeholder **no** reemplaza al label.
- Agrupar campos relacionados con `<fieldset>`/`<legend>`.
- Errores con texto claro, asociados al campo (`aria-describedby`), no solo color rojo.
- Campos requeridos indicados visualmente y con `required`.

## Color y contraste
- Texto normal: contraste mínimo 4.5:1. Texto grande: 3:1.
- Verificar los pares de color del theme contra el fondo. Si el cliente da una paleta con bajo contraste, avisar y proponer ajuste.

## Estructura y navegación
- Landmarks correctos (`header`, `nav`, `main`, `footer`) — uno de `main` por página.
- Jerarquía de headings sin saltos (ver `html-semantics.md`).
- "Skip link" al inicio para saltar al `<main>` en sitios con nav larga.
- Si hay varias `<nav>`, distinguirlas con `aria-label`.

## ARIA: solo cuando hace falta
- **Primera regla de ARIA: no usar ARIA si un elemento HTML nativo hace el trabajo.**
- Usar ARIA para patrones sin equivalente nativo (menús complejos, tabs, modales): roles y estados (`aria-expanded`, `aria-controls`, `aria-current`).
- No agregar roles redundantes (`role="button"` en un `<button>` está de más).

## Movimiento
- Respetar `prefers-reduced-motion`: reducir o quitar animaciones para quien lo pide.

## Para auditar
Servir el sitio y correr una auditoría de accesibilidad (axe-core en runtime). Recordar: las herramientas automáticas detectan solo una parte de los problemas reales; complementar con prueba manual de teclado y, si es posible, lector de pantalla.
